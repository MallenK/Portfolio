API REST: buenas prácticas desde el backend

19 septiembre 2025 · Lectura: 4 min

Una API útil debe ser predecible, segura y observable. Estas son las reglas que sigo para lanzar y evolucionar servicios REST sin sorpresas.

1. Diseña recursos consistentes

  • Utiliza nombres en plural y evita verbos en la URL (/users, /orders/{id}).
  • Mantén filtros y paginación coherentes (?page=, ?per_page=, ?sort=).
  • Incluye HATEOAS cuando el cliente lo requiera (links self, next, prev).

2. Versiona de forma explícita

Versionar desde el inicio reduce fricción cuando rompes compatibilidad.

  • Prefiero versionar en la URL (/api/v1/) para visibilidad inmediata.
  • Mantén políticas de deprecación claras y notifica con antelación mediante headers Sunset.
  • Automatiza pruebas contractuales por versión para evitar regresiones silenciosas.

3. Devuelve respuestas ricas y códigos HTTP correctos

Escenario Código Cuerpo recomendado
Creación exitosa 201 Created { "id": "uuid", "resource": { ... } }
Validación fallida 422 Unprocessable Entity { "errors": [{ "field": "email", "message": "Formato inválido" }] }
Error interno controlado 500 Internal Server Error { "traceId": "uuid", "message": "Intenta más tarde" }

4. Seguridad y observabilidad

  • Implementa OAuth2 o JWT firmados y rota claves periódicamente.
  • Registra métricas en Prometheus/OpenTelemetry (latencia, throughput, errores).
  • Rate limiting distribuido con Redis o API Gateway para frenar abusos.
  • Documenta scopes y permisos en Swagger u OpenAPI.

5. Checklist de despliegue

  1. Genera changelog automatizado con los endpoints nuevos/obsoletos.
  2. Valida migraciones y seeders en staging con datos reales.
  3. Ejecuta pruebas contractuales (Pact) contra los consumidores clave.
  4. Actualiza SDKs/clients y comunícalo a los integradores.
